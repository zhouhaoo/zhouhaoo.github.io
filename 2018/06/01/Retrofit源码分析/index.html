Android Retrofit 笔记 网络请求 源码分析

Retrofit源码分析

"最好用的android网络请求框架的封装"

Posted by zhouhaoo on 2018年06月01日

APP通过Retrofit请求网络，实际上是使用Retrofit接口层封装请求参数，之后由OkHTTP完成后续的请求操作，然后在服务端返回数据后，OkHTTP将原始的结果交给Retrofit，Retrofit根据我们的需求对结果进行解析。本篇主要分析Retrofit 2.3.0的封装，okhttp分析见okhttp源码分析

Retrofit的基本使用

  1. 创建Retrofit对象

    Retrofit retrofit = new Retrofit.Builder()
    .baseUrl("https://api.github.com")
    .addConverterFactory(GsonConverterFactory.create())
    .build();

  2. 创建service接口

    public interface GitHubService {
  @GET("users/{user}/repos")
  Call<List<Repo>> listRepos(@Path("user") String user);
}
...
GitHubService service = retrofit.create(GitHubService.class);

  3. 发起请求

    Call<List<Repo>> repos = service.listRepos("octocat");
repos.enqueue....//异步请求

以上三步是Retrofit官网上使用的简易步骤，当然还可以和Rxjava结合等等。

  • 利用builder构建者模式创建一个Retrofit对象

  • 上面的GitHubService是创建了一个接口对象，用在接口方法上注解的方式，表明了请求方式、请求URL、请求参数等等，这也是后面retrofit利用动态代理实现的基础。
  • 执行请求。

创建Service：

public <T> T create(final Class<T> service) {
    Utils.validateServiceInterface(service);
    if (validateEagerly) {
      eagerlyValidateMethods(service);
    }
    return (T) Proxy.newProxyInstance(service.getClassLoader(), new Class<?>[] { service },
        new InvocationHandler() {
          private final Platform platform = Platform.get();

          @Override public Object invoke(Object proxy, Method method, @Nullable Object[] args)
              throws Throwable {
            // If the method is a method from Object then defer to normal invocation.
            if (method.getDeclaringClass() == Object.class) {
              return method.invoke(this, args);
            }
            if (platform.isDefaultMethod(method)) {
              return platform.invokeDefaultMethod(method, service, proxy, args);
            }
             //创建serviceMethod对象
            ServiceMethod<Object, Object> serviceMethod =
                (ServiceMethod<Object, Object>) loadServiceMethod(method);
              //okHttpCall
            OkHttpCall<Object> okHttpCall = new OkHttpCall<>(serviceMethod, args);
              //返回转换好的值
            return serviceMethod.callAdapter.adapt(okHttpCall);
          }
        });
  }

简单点来讲 ，当调用Create方法的时候，就是利用动态代理实例化出一个Service对象，当对象调用定义好的接口方法的时候，通过解析得到一个ServiceMethod对象，ServiceMethod内部维护了请求的参数，紧接着创建出一个OkHttpCall对象，解析出对应的Call封装，最后得到一个我们之前对应的接口返回值，最终请求是转交给OKhttp执行的。下面详细分析对应的流程。

Retrofit的成员变量

在Retrofit内部有七个成员变量，分别来看看什么含义：

public final class Retrofit {
  private final Map<Method, ServiceMethod<?, ?>> serviceMethodCache = new ConcurrentHashMap<>();
  final okhttp3.Call.Factory callFactory;
  final HttpUrl baseUrl;
  final List<Converter.Factory> converterFactories;
  final List<CallAdapter.Factory> adapterFactories;
  final @Nullable Executor callbackExecutor;
  final boolean validateEagerly;
    ....
}
  • serviceMethodCache:一个线程安全的map，用于缓存对象的ServiceMethod
  • callFactory:call工厂，可自定义Okhttp配置
  • baseUrl:请求基地址，用于拼接Service上的注解url
  • converterFactories:转换工厂list，用于不同的对象转换，例如GsonConverterFactory等
  • adapterFactories:数据解析工厂list，解析请求响应的数据，默认Gson
  • callbackExecutor:回调执行器，切换异步回调的线程，请求结束后，回调主线程。
  • validateEagerly:标志位，用于Retrofit的create方法

核心重点就在retrofit中的Create方法中的三行代码了：

ServiceMethod<Object, Object> serviceMethod = (ServiceMethod<Object, Object>) loadServiceMethod(method);
OkHttpCall<Object> okHttpCall = new OkHttpCall<>(serviceMethod, args);
return serviceMethod.callAdapter.adapt(okHttpCall);

serviceMethod

先来看看loadServiceMethod这个方法 ：

 ServiceMethod<?, ?> loadServiceMethod(Method method) {
    ServiceMethod<?, ?> result = serviceMethodCache.get(method);
    if (result != null) return result;

    synchronized (serviceMethodCache) {
      result = serviceMethodCache.get(method);
      if (result == null) {
          //创建ServiceMethod对象
        result = new ServiceMethod.Builder<>(this, method).build();
        serviceMethodCache.put(method, result);
      }
    }
    return result;
  }

这个的主要功能是从serviceMethodCache 缓存中去取缓存，在保证线程安全的情况下，假如没有缓存对象，则去创建一个serviceMethod对象，并将其加入到缓存map中。继续跟代码，看看serviceMethod的创建：

  1. callAdapter = createCallAdapter();
       private CallAdapter<T, R> createCallAdapter() {
         Type returnType = method.getGenericReturnType();
         if (Utils.hasUnresolvableType(returnType)) {
           throw methodError(
               "Method return type must not include a type variable or wildcard: %s", returnType);
         }
         if (returnType == void.class) {
           throw methodError("Service methods cannot return void.");
         }
         Annotation[] annotations = method.getAnnotations();
         try {
           //noinspection unchecked
           return (CallAdapter<T, R>) retrofit.callAdapter(returnType, annotations);
         } catch (RuntimeException e) { // Wide exception range because factories are user code.
           throw methodError(e, "Unable to create call adapter for %s", returnType);
         }
       }

callAdapter 最终调用了nextCallAdapter这个方法，上面逻辑就是从初始化Retrofit的adapterFactories中，根据returnType从adapterFactories获取对应的CallAdapter，在android平台默认的情况下，这个Factory就是ExecutorCallAdapterFactory，依据是在Retrofit初始化有个设置，见以下代码：

           throw new IllegalStateException("Base URL required.");

         if (callFactory == null) {
           callFactory = new OkHttpClient();
         }

         if (callbackExecutor == null) {

说明默认情况下最终createCallAdapter()获取到的就是这个ExecutorCallAdapterFactory,

   final class ExecutorCallAdapterFactory extends CallAdapter.Factory {
     final Executor callbackExecutor;
   
     ExecutorCallAdapterFactory(Executor callbackExecutor) {
       this.callbackExecutor = callbackExecutor;
     }
   
     @Override
     public CallAdapter<?, ?> get(Type returnType, Annotation[] annotations, Retrofit retrofit) {
       if (getRawType(returnType) != Call.class) {
         return null;
       }
       final Type responseType = Utils.getCallResponseType(returnType);
       return new CallAdapter<Object, Call<?>>() {
         @Override public Type responseType() {
           return responseType;
         }
   
         @Override public Call<Object> adapt(Call<Object> call) {
           return new ExecutorCallbackCall<>(callbackExecutor, call);
         }
       };
     }
   }

看看上面代码，当我们创建的Servicemethod返回值是Call的时候，就会执行这个Factory的get方法，返回这个用于android平台，异步调用的CallAdapter。至此，createCallAdapter分析完毕。

OkHttpCall

final class OkHttpCall<T> implements Call<T> {
  private final ServiceMethod<T, ?> serviceMethod;
  private final @Nullable Object[] args;

  private volatile boolean canceled;

  @GuardedBy("this")
  private @Nullable okhttp3.Call rawCall;
  @GuardedBy("this")
  private @Nullable Throwable creationFailure; // Either a RuntimeException or IOException.
  @GuardedBy("this")
  private boolean executed;
//创建OkHttpCall对象
  OkHttpCall(ServiceMethod<T, ?> serviceMethod, @Nullable Object[] args) {
    this.serviceMethod = serviceMethod;
    this.args = args;
  }
}

所以，下面举个异步调用的例子：

 @Override public void enqueue(final Callback<T> callback) {
      checkNotNull(callback, "callback == null");

      delegate.enqueue(new Callback<T>() {
        @Override public void onResponse(Call<T> call, final Response<T> response) {
           //利用post的callbackExecutor将回调切回主线程。
          callbackExecutor.execute(new Runnable() {
            @Override public void run() {
              if (delegate.isCanceled()) {
                callback.onFailure(ExecutorCallbackCall.this, new IOException("Canceled"));
              } else {
                callback.onResponse(ExecutorCallbackCall.this, response);
              }
            }
          });
        }

        @Override public void onFailure(Call<T> call, final Throwable t) {
          callbackExecutor.execute(new Runnable() {
            @Override public void run() {
              callback.onFailure(ExecutorCallbackCall.this, t);
            }
          });
        }
      });
    }

当有一个网络请求正式执行开始后，在Retrofit默认配置的情况，异步执行是在这里！！！，这里用了一个静态代理的设计模式，上面那个delegate对象，实际上就是OkHTTPCall对象了，首先看看这个callbackExecutor对象,这个就是在Retrofit初始化时候赋值的，对应的就是Platform类中的Android平台，这里用到了一个handler并且了使用的是MainLooper，最终execute方法对应的就是handler.post(),根据handler的机制，这个Runnable中的run方法将执行在主线程，这里也解释了Retrofit为何能在异步请求的时候，能自动帮助我们切换线程的原因：

//Platform.java  
static class Android extends Platform {
    @Override public Executor defaultCallbackExecutor() {
      return new MainThreadExecutor();
    }

    @Override CallAdapter.Factory defaultCallAdapterFactory(@Nullable Executor callbackExecutor) {
      if (callbackExecutor == null) throw new AssertionError();
      return new ExecutorCallAdapterFactory(callbackExecutor);
    }

    static class MainThreadExecutor implements Executor {
      private final Handler handler = new Handler(Looper.getMainLooper());

      @Override public void execute(Runnable r) {
        handler.post(r);
      }
    }
  }

所以这里通过callbackExecutor执行后，最终就在主线程啦，根据代理对象OkhtppCall，真正enqueue是他：

 @Override public void enqueue(final Callback<T> callback) {
    checkNotNull(callback, "callback == null");
	//call 为okhttp的call 证明实际上是调用okhttp啦
    okhttp3.Call call;
    Throwable failure;

    synchronized (this) {
      if (executed) throw new IllegalStateException("Already executed.");
      executed = true;

      call = rawCall;
      failure = creationFailure;
      if (call == null && failure == null) {
        try {
            //1.创建call对象，用okhttp请求
          call = rawCall = createRawCall();
        } catch (Throwable t) {
          failure = creationFailure = t;
        }
      }
    }

    if (failure != null) {
      callback.onFailure(this, failure);
      return;
    }

    if (canceled) {
      call.cancel();
    }
	//2.发起异步请求
    call.enqueue(new okhttp3.Callback() {
      @Override public void onResponse(okhttp3.Call call, okhttp3.Response rawResponse)
          throws IOException {
        Response<T> response;
        try {
          //3. 解析响应
          response = parseResponse(rawResponse);
        } catch (Throwable e) {
          callFailure(e);
          return;
        }
        callSuccess(response);
      }

      @Override public void onFailure(okhttp3.Call call, IOException e) {
        try {
          callback.onFailure(OkHttpCall.this, e);
        } catch (Throwable t) {
          t.printStackTrace();
        }
      }

      private void callFailure(Throwable e) {
        try {
          callback.onFailure(OkHttpCall.this, e);
        } catch (Throwable t) {
          t.printStackTrace();
        }
      }

      private void callSuccess(Response<T> response) {
        try {
          callback.onResponse(OkHttpCall.this, response);
        } catch (Throwable t) {
          t.printStackTrace();
        }
      }
    });
  }

  3. 解析响应: response = parseResponse(rawResponse);

    Response<T> parseResponse(okhttp3.Response rawResponse) throws IOException {
    ResponseBody rawBody = rawResponse.body();
   
    // Remove the body's source (the only stateful object) so we can pass the response along.
    rawResponse = rawResponse.newBuilder()
        .body(new NoContentResponseBody(rawBody.contentType(), rawBody.contentLength()))
        .build();
   
    int code = rawResponse.code();
    if (code < 200 || code >= 300) {
      try {
        // Buffer the entire body to avoid future I/O.
        ResponseBody bufferedBody = Utils.buffer(rawBody);
        return Response.error(bufferedBody, rawResponse);
      } finally {
        rawBody.close();
      }
    }
   
    if (code == 204 || code == 205) {
      rawBody.close();
      return Response.success(null, rawResponse);
    }
   
    ExceptionCatchingRequestBody catchingBody = new ExceptionCatchingRequestBody(rawBody);
    try {
        //转换响应对象
      T body = serviceMethod.toResponse(catchingBody);
      return Response.success(body, rawResponse);
    } catch (RuntimeException e) {
      // If the underlying source threw an exception, propagate that rather than indicating it was
      // a runtime exception.
      catchingBody.throwIfCaught();
      throw e;
    }
  }

    上面解析响应的上半部分代码只是做了一个响应的判断，创建Response对象，关键是toResponse这个方法：

    //ServiceMethod.java  
R toResponse(ResponseBody body) throws IOException {
    return responseConverter.convert(body);
  }

    responseConverter之前已经根据ServiceMethod中判断过了 ，这里这个就是用responseConverter来得到我们指定返回类型的响应body，至此完成这个请求。

流程步骤总结

  1. 创建Retrofit实例对象 （包含了一些默认的配置）
  2. 定义一个网络请求接口，并为接口中的方法添加注解
  3. 通过动态代理生成网络请求对象（创建对应的callAdapter和converter）
  4. 通过网络请求适配器将网络请求对象进行平台适配
  5. 通过网络请求执行器发送网络请求
  6. 通过数据转换器解析数据
  7. 通过回调执行器切换线程（handler&静态代理）
  8. 用户在主线程处理返回结果

总结

Retrofit和Okhttp相比，代码量和逻辑相对简单些，但是里面有特别多的设计模式：比如代理模式（动态和静态代理都有），工厂模式，外观模式，策略模式，适配器模式等等，利用这些，实现了一个高度可配置，简易使用的网络请求框架的封装。
